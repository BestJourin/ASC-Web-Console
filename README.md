# Sivy ASC Web Console

This public repository hosts the standalone Web Bluetooth console for Sivy ASC bring-up, register tests, ADC sampling display, and MCUboot/MCUmgr BLE OTA.

The page is static. It does not contain firmware source code, signing keys, build outputs, or hardware-private files.

## Public URL

After GitHub Pages is enabled, the expected URL is:

```text
https://bestjourin.github.io/ASC-Web-Console/
```

If GitHub shows a different Pages URL, use the URL from the repository Pages settings or the `Deploy Pages` workflow output.

## How BLE Access Works

Web Bluetooth always uses the Bluetooth adapter of the computer that opens the page.

| Scenario | Result |
| --- | --- |
| A tester opens the HTTPS page and connects an ASC board near their computer | Supported |
| A remote tester opens the page and tries to connect a board near your computer | Not supported |
| The page is opened from `https://...` | Web Bluetooth can be available |
| The page is opened from `http://localhost:8080` | Web Bluetooth can be available |
| The page is opened from `http://192.168.x.x:8080` or by double-clicking `index.html` | Web Bluetooth is disabled |

Use desktop Chrome or Edge. iOS Safari does not support Web Bluetooth.

## Local Development

Run a local static server:

```powershell
python -m http.server 8080
```

Open:

```text
http://localhost:8080
```

The top status bar shows:

```text
Page origin
Secure context
Web Bluetooth
BLE adapter
```

The connect button is enabled only when the browser provides a secure context and Web Bluetooth.

## Device Selection

The page offers three scan modes:

| Mode | Use |
| --- | --- |
| All devices | Broad bring-up mode. The browser chooser displays nearby connectable BLE devices. |
| Name prefix | Filters by the configured prefix, default `Sivy_ASC`. |
| Target service | Uses ASC/SMP service UUIDs and known names as filters. |

If the target board is not shown in the browser chooser, make sure it is advertising and not already connected to a phone, nRF Connect, or another computer.

## OTA Files

Supported OTA inputs:

```text
zephyr.signed.bin
dfu_application.zip
```

For the ASC firmware project, these are normally generated by the private firmware repository under:

```text
firmware/app/build/app/zephyr/zephyr.signed.bin
firmware/app/build/dfu_application.zip
```

Do not upload `merged.hex` through Web OTA. `merged.hex` is for first-time J-Link / programmer flashing, not BLE OTA.

## OTA Image State

The console reads MCUboot image state after upload and marks the non-active image for test boot by image hash. It accepts MCUboot image hashes of 32, 48, or 64 bytes. ED25519/SHA512 configurations commonly return `len=64`.

If upload reaches 100% but the page reports:

```text
No non-active bootable image hash found in image state
```

first force-refresh the page and check that the event log reports a current Web Console build. If the secondary slot hash matches the active slot hash, the image is the same as the running firmware and no reboot swap can be proven.

## Pages Deployment

This repository contains a GitHub Actions workflow:

```text
.github/workflows/pages.yml
```

Enable:

```text
Settings -> Pages -> Source -> GitHub Actions
```

Then run or push to `main`. The workflow uploads the repository root as the Pages artifact.
